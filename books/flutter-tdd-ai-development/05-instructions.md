---
title: "第5章 AIに「チームの流儀」を教える ― instructions ファイル設計"
---

第4章では、テストの解像度がそのままAIの実装の解像度になることを確認しました。しかし、テストだけでは伝えきれないものがあります。ディレクトリ構成、命名の慣習、レイヤーの切り方といった「チームの流儀」です。本章では、それらをAIに伝えるための instructions ファイルを設計します。

## テストは個別仕様、instructionsは全体規約

まず、本書の軸となる対比を明確にしておきます。

**テストは個別の仕様書**です。「increment で value が 1 増える」というテストは、その1つのふるまいについて、入力と期待結果を厳密に定義します。しかしテストは「どのディレクトリにファイルを置くか」「クラスをどう命名するか」までは語りません。

**instructions ファイルは全体の規約書**です。プロジェクト全体に共通するアーキテクチャ、命名規則、やってはいけないことを1か所にまとめ、AIコーディングツールがすべての応答の前提として読み込むものです。

人間のチーム開発に例えるなら、テストはチケットごとの受け入れ条件、instructions ファイルはコーディング規約とアーキテクチャドキュメントに相当します。新しくジョインしたメンバーに受け入れ条件だけ渡しても、そのチームらしいコードは書けません。逆に規約だけ渡しても、何を作ればいいのか分かりません。両方が揃って初めて「チームの一員として動ける」のは、AIも同じです。

第4章の実験を思い出してください。良いテストを渡せばAIは正しいロジックに収束しました。しかし、生成されたコードをどこに置くか、テスト名をどう付けるかは、依頼のたびに揺れます。この「揺れ」を抑えるのが instructions ファイルの役割です。

## copilot-instructions.md / CLAUDE.md に何を書くか

instructions ファイルの実体は、ツールごとに置き場所が決まった Markdown ファイルです。

- **GitHub Copilot**: リポジトリの `.github/copilot-instructions.md`
- **Claude Code**: リポジトリ直下の `CLAUDE.md`

さらに近年は、特定のツールに依存しない共通規格として **AGENTS.md** をリポジトリ直下に置く動きも広がっています。複数のAIコーディングツールを併用するチームでは、規約の本体を1つのファイルに集約し、各ツール用のファイルからそれを参照させる構成が現実的です。本書のサンプルは copilot-instructions.md / CLAUDE.md のどちらに置いても機能する内容にしてあります。

書くべき内容は、大きく4つです。

1. **プロジェクト概要**: 何のアプリで、主要な技術選定は何か
2. **アーキテクチャ規約**: ファイルをどこに置き、レイヤーをどう分けるか
3. **テスト規約**: テストの置き場所、命名、構造、モックの方針
4. **やってはいけないこと**: 禁止事項の明文化

冒頭の2セクションは、本書のプロジェクトでは次のようになります（完全版は付録Bに掲載します。本章では骨子を抜粋しながら解説します）。

```md
# プロジェクト規約（copilot-instructions.md / CLAUDE.md 共通）

## プロジェクト概要
- Flutter 製カウンターアプリ。状態管理は BLoC（flutter_bloc + equatable）。
```

概要はこの程度の短さで十分です。重要なのは「状態管理は BLoC」のような**技術選定を明記する**ことです。これがないと、AIは依頼のたびに setState / Provider / Riverpod などから「それらしいもの」を選び直してしまい、コードベースに複数の流儀が混在していきます。なお、本書のカウンターは現時点ではまだ setState のままですが、第6章で行う Bloc への移行を見据えて、目指すアーキテクチャを先に規約として宣言しています。

:::message
instructions ファイルは「常に読み込まれる」前提のため、長くしすぎると重要な規約が埋もれます。Claude Code には、特定の作業のときだけ参照される手順書を Skills として切り出す仕組みがあり、「常に守る規約は CLAUDE.md、必要なときだけ開く詳細手順は Skills」という分担ができます。詳細は公式ドキュメントを参照してください。
:::

## アーキテクチャ規約を伝える：レイヤー分離と依存の方向

アーキテクチャ規約のセクションを見てみます。

```md
## アーキテクチャ規約
- 機能ごとに lib/<feature>/ を切る。UI はロジックを持たない。
- Event / State / Bloc は1つの <feature>_bloc.dart にまとめる。
- UI は Bloc の add を直接呼ばず、Bloc が公開するメソッド（add をラップした入口）を呼ぶ。
```

3行しかありませんが、それぞれがAIの典型的な迷いどころを1つずつ潰しています。

1行目は**レイヤー分離と依存の方向**です。「UI はロジックを持たない」と書くことで、第2章で見た「_counter と setState が Widget 内に同居した状態」への逆戻りを防ぎます。ロジックは lib/<feature>/ 配下のクラスに置き、UI はそれを参照するだけ、という依存の向きが固定されます。

2行目は**ファイル構成の粒度**です。BLoC の流儀にはファイルを細かく分ける派とまとめる派があり、規約がなければAIの出力はどちらにも振れます。どちらが正しいかではなく、「このチームではこう」と決めてあることが重要です。

3行目は**呼び出し口の統一**です。UI から `bloc.add(const CounterIncremented())` と書くか `bloc.increment()` と書くかも、規約がなければ揺れます。本書では公開メソッド経由に統一します。この規約の効果は、第6章で Bloc を実装するときに具体的に確認します。

いずれも共通するのは、**選択肢が複数ある場所にだけ規約を書く**という原則です。Dart の言語仕様で一意に決まることを書いても文字数の無駄です。AIが（そして新メンバーが）迷う分岐点だけを狙って書きます。

## テスト規約を伝える：命名、given/when/then、モックの方針

続いてテスト規約と禁止事項です。

```md
## テスト規約
- テストは lib/ と同じ構成で test/ に置く。
- テスト名は日本語で「<操作> で <結果>」。
- 構造は given / when / then コメントで区切る。
- 自作クラスはモックしない。モックするのは外部境界（リポジトリ等）だけ。

## やってはいけないこと
- テストを実装に合わせて書き換えて通さない（テストが仕様）。
- 依頼されていないファイルを変更しない。
```

テスト名の「<操作> で <結果>」形式は、第3章で書いた「increment で value が 1 増える」そのものです。テスト名を仕様文として読める形に揃えておくと、テスト一覧がそのまま仕様書の目次になります。given / when / then のコメント区切りも同様に、第3章のテストコードで実践した構造をそのまま規約化したものです。**すでにコードベースにある良い実例を規約として明文化する**のが、instructions ファイル作りの近道です。

モックの方針は第8章で外部依存を扱うときに効いてくる規約ですが、先に書いておく価値があります。AIは「テストしやすくするため」と称して自作クラスまで手当たり次第にモック化する傾向があり、放置するとテストが実装の写し絵になって壊れやすくなるためです。

そして「やってはいけないこと」の1行目は、本書のコアメッセージの裏返しです。テストが仕様である以上、**実装に合わせてテストを書き換えて通すのは仕様の改ざん**です。AIは「テストを通す」ことを目標に与えられると、テスト側を修正するという最短経路を選ぶことがあります。この一線だけは、規約として明文的に封じておきます。

## 規約×テストが揃ったときのAIの出力精度を確認する

それでは、規約とテストの両方が揃った状態で、第4章と同じ依頼をやり直してみます。instructions ファイルをリポジトリに置いたうえで、良いテスト（初期値0 / 1回で1 / 2回で2 の3ケース）を渡し、次のように依頼します。

> test/counter/counter_test.dart のテストがすべて通るように Counter クラスを実装してください。

第4章との違いは、依頼文には書いていない情報をAIが規約から補うことです。典型的には次のような挙動になります。

- ファイルを `lib/counter/counter.dart` に作る（「機能ごとに lib/<feature>/ を切る」に従う）
- Counter クラスだけを作り、main.dart など依頼していないファイルには触れない
- テストコードを一切変更しない

さらに、実装ではなく**テストの追加**を依頼したときの差はより顕著です。

> Counter に「連続して increment しても値が壊れない」ことを確認するテストを1つ追加してください。

規約がない状態では、テスト名が英語になったり、AAA コメント（Arrange / Act / Assert）で区切られたりと、既存テストと流儀の違うコードが返ってきがちです。規約がある状態では、日本語の「<操作> で <結果>」形式の名前と given / when / then 区切りで、既存の3ケースと見分けのつかないテストが返ってくるのが典型例です。生成されたコードのレビューで「動くかどうか」だけでなく「うちの流儀かどうか」を毎回指摘していた手間が、規約を1度書くことで先回りして消える、というのが instructions ファイルの投資対効果です。

:::message
instructions ファイルは書いて終わりではありません。AIの出力に同じ種類の逸脱が繰り返し見つかったら、それは規約に追記すべきシグナルです。レビュー指摘を規約に還元するサイクルを回すことで、ファイルはチームの実態に沿って育っていきます。
:::

テストという個別仕様と、instructions という全体規約。この2つが揃ったことで、AIに安心して任せられる範囲が大きく広がりました。次章では、この体制を使って、いよいよカウンターを Bloc ベースの状態管理に作り直します。規約に先行して宣言した「状態管理は BLoC」を、テストファーストで実現していきます。
