title: "付録C よくあるAIの失敗パターンと対処法"
---

本書の各章で遭遇した、あるいは遭遇しやすいものとして紹介した AI の失敗パターンを一覧にまとめます。いずれも特定のツールの欠陥ではなく、GitHub Copilot でも Claude Code でも起こりうる「傾向」です。症状に気づいたら、原因を特定し、テストか instructions ファイルのどちらかに対処を刻み込むのが基本方針です。

## 失敗パターン一覧表

| # | 失敗パターン | 症状 | 原因 | 対処 |
| --- | --- | --- | --- | --- |
| 1 | テストだけ通す過剰適合 | `int get value => 1;` のような定数返しでテストが緑になる。動かすと明らかに仕様と違う | テストの解像度が低い（1ケースだけ、greaterThan などの緩い検証）ため、最小実装が「ズル」で成立してしまう | ケースを増やして三角測量する（初期値 0 / 1回で 1 / 2回で 2）。値の検証は具体的な期待値で書く |
| 2 | モックし過ぎ | 自作の State や値オブジェクトまで Mock クラスが生成され、テストが実装の写しになる。リファクタリングのたびにテストが壊れる | 「テスト＝全部モック」という誤った一般化。モックの目的（外部境界の切り離し）が伝わっていない | instructions ファイルに「自作クラスはモックしない。モックするのは外部境界（リポジトリ等）だけ」と明記する。レビューで Mock の数を確認する |
| 3 | props への追加漏れ | State にフィールドを追加したのに blocTest の expect が通らない、あるいは逆に「値が違うのに等しい」と判定されて Red になるべきテストが緑のまま | Equatable の等価判定は props に列挙したフィールドだけで行われる。AI はフィールド追加時に props の更新を忘れることがある | 追加フィールドが異なる2つの State を比較するテストを書く。instructions ファイルに「フィールドを追加したら必ず props にも追加する」と明記する |
| 4 | async に thenReturn | mocktail のスタブで実行時エラーが出る、または Future の解決を待つテストが不安定になる | `Future<int?> load()` のような async メソッドに thenReturn を使っている。Future を返すメソッドは thenAnswer((_) async => 値) で書く必要がある | instructions ファイルに「Future を返すメソッドのスタブは thenAnswer で書く」と明記する。setUp でのデフォルトスタブをテンプレ化して使い回す |
| 5 | 頼んでいないリファクタの拡大 | 「このテストを通して」と頼んだだけなのに、無関係なファイルのリネーム・整形・構造変更まで差分に含まれる | AI は「ついでに良くする」方向へ寄りやすく、依頼の範囲を自分では線引きしない | 依頼文で対象ファイルと変更範囲を明示する。instructions ファイルに「依頼されていないファイルを変更しない」と明記し、差分レビューで範囲外の変更を差し戻す |
| 6 | 存在しないAPIの使用 | 一見それらしいメソッドやパラメータ（実在しない blocTest の引数、旧バージョンの API など）を使ったコードが生成され、コンパイルが通らない | 学習データ上の複数バージョン・類似ライブラリの API が混ざる。バージョン差異は AI の弱点になりやすい | flutter analyze と flutter test を必ず実行して機械的に検出する。pubspec.yaml のバージョンを instructions ファイルやプロンプトで伝え、公式ドキュメントの API と突き合わせる |
| 7 | テストを実装に合わせて書き換える | 「テストが通りません」と伝えたら、実装ではなく expect の期待値のほうが修正されて緑になる | AI にとって「テストを直す」ことも「緑にする」解の1つであり、どちらが仕様かを指定しないと区別できない | instructions ファイルに「テストを実装に合わせて書き換えて通さない（テストが仕様）」と明記する。依頼時にも「テストは変更禁止」と添える。CI でテストファイルの差分を確認する |
| 8 | 初期状態と同一の State を expect に書く | 「何も起きない」ことを検証したいのに `expect: () => [CounterState()]` と書かれ、テストが落ち続ける | 現在の state と等しい State は、一度でも emit された後は再 emit されない（ただしインスタンス最初の emit は初期状態と等価でも通知される）。この挙動を知らないと期待リストを間違える | 「何も emit されない」ことは空リスト `const <CounterState>[]` で検証する。この挙動自体を instructions ファイルのテスト規約に書いておく |

## 対処の共通原則

表の対処欄を見渡すと、行き着く先は2つしかないことが分かります。1つは**テストの解像度を上げること**（パターン1・3・8）、もう1つは**規約を instructions ファイルに刻むこと**（パターン2・4・5・7）です。そして両者の外側を flutter analyze・flutter test・CI という機械的なゲート（パターン6）が支えます。

失敗はゼロにはなりません。しかし、失敗のたびにテストか規約が1行育つなら、それはチームにとっての資産です。AI の失敗を「またか」で終わらせず、仕様と規約に翻訳し続けること。それが本書で扱ったテスト駆動AI開発の、日々の運用における結論です。
