第5章では骨子の抜粋と設計意図を解説しました。ここでは、copilot-instructions.md と CLAUDE.md（および AGENTS.md）のどれに置いても使える完全版テンプレートを示します。プロジェクトに合わせて固有名詞を置き換えれば、そのまま使い始められます。

## テンプレート全文

```md
# プロジェクト規約（copilot-instructions.md / CLAUDE.md 共通）

## プロジェクト概要
- Flutter 製カウンターアプリ。状態管理は BLoC（flutter_bloc + equatable）。
- パッケージ名は counter_app（import 'package:counter_app/...'）。
- Dart 3 以降を前提とし、sealed class を使用する。
- 永続化は shared_preferences を CounterRepository 経由で使う。

## アーキテクチャ規約
- 機能ごとに lib/<feature>/ を切る。UI はロジックを持たない。
- Event / State / Bloc は1つの <feature>_bloc.dart にまとめる。
- UI は Bloc の add を直接呼ばず、Bloc が公開するメソッド（add をラップした入口）を呼ぶ。
- Event クラスは sealed class を継承し、const コンストラクタを持たせる。
- State は Equatable を継承した不変クラスにし、フィールドを追加したら必ず props にも追加する。
- State の部分更新には copyWith を使う。ただし copyWith は必要になってから導入する。
- 外部サービス（ストレージ・ネットワーク等）へのアクセスは abstract class の
  リポジトリで抽象化し、Bloc にはコンストラクタで注入する。

## テスト規約
- テストは lib/ と同じ構成で test/ に置く。
- テスト名は日本語で「<操作> で <結果>」。
- 構造は given / when / then コメントで区切る。
- Bloc のテストは bloc_test パッケージの blocTest を使う。
  - build: には Bloc を生成する関数を渡す。
  - expect: は emit される State のリストで書く。現在の state と等しい State は、
    一度でも emit された後は再 emit されない（ただしインスタンス最初の emit は
    初期状態と等価でも通知される）ことに注意する。
- モックには mocktail を使う。
  - 自作クラスはモックしない。モックするのは外部境界（リポジトリ等）だけ。
  - Future を返すメソッドのスタブは thenAnswer で書く（thenReturn は使わない）。
  - verify は「仕様として保証したい呼び出し」だけに絞り、実装詳細を縛らない。

## 実装の進め方
- テストが先、実装が後。渡されたテストが Red であることを確認してから実装する。
- テストを通す最小限の実装から始め、その後にリファクタリングする。
- 実装後は flutter test を実行し、全テストが緑であることを確認する。
- dart format と flutter analyze を通してから完了とする。

## やってはいけないこと
- テストを実装に合わせて書き換えて通さない（テストが仕様）。
- 依頼されていないファイルを変更しない。
- 依頼されていないリファクタリングや機能追加をしない。
- pubspec.yaml に無断で依存パッケージを追加しない。
- print デバッグや TODO コメントを成果物に残さない。
```

## 運用のポイント

テンプレートはあくまで出発点です。第5章で述べたとおり、instructions ファイルは「AI が同じ失敗を繰り返したら1行足す」という運用で育てるものです。最初から網羅を目指すより、実際に起きたズレを規約に翻訳していくほうが、チームの実態に合った文書になります。

また、「やってはいけないこと」の節は効果が高い一方で、増やしすぎると個々の禁止事項が薄まる傾向があります。数カ月に一度は見直し、アーキテクチャ規約やテスト規約として一般化できるものは移動させることをおすすめします。

このテンプレートと付録Aのテスト群がそろえば、AI との協働の土台は完成です。最後の付録Cでは、それでも起きる典型的な失敗と対処をまとめます。
