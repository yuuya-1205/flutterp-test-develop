title: "第9章 CIで回す ― テストがゲートになる開発フロー"

第8章までで、カウンターアプリはロジック分離・Bloc 化・機能拡張・永続化と進化し、その全過程をテストが支えてきました。しかし、テストはローカルで手動実行しているだけでは力を半分しか発揮できません。本章では GitHub Actions でテストを自動実行し、「テストが通らないコードはマージされない」という**ゲート**を作ります。

## GitHub Actions でテストを自動実行する

Pull Request が作られるたびにテストが走るワークフローを定義します。リポジトリのルートに以下のファイルを置くだけです。

```yaml
# .github/workflows/ci.yml
name: CI

on:
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          channel: stable
      - run: flutter pub get
      - run: dart format --output=none --set-exit-if-changed .
      - run: flutter analyze
      - run: flutter test --coverage
```

チェックは3段構えで、軽いものほど先に置くのが定石です。

1. **`dart format --output=none --set-exit-if-changed .`** ― フォーマット違反があれば即失敗し、整形の議論を人間のレビューから追放できます。
2. **`flutter analyze`** ― 静的解析で、未使用の import や型の不整合をテスト以前に落とします。
3. **`flutter test --coverage`** ― 本書のすべてのテストを実行します。第2章から一度も書き換えていないデフォルトの `widget_test.dart` も、ここで毎回検証されます。カバレッジレポートも同時に生成されます（使い方は後述）。

## 「AIが書いたコードはCIを通るまでマージしない」ルール

CI が整ったら、チームにひとつだけルールを追加します。

**AI が書いたコードは、CI を通るまでマージしない。人間が書いたコードと同じ基準で。**

当たり前に聞こえますが、現場では「AI が書いたから大丈夫だろう」「差分が大きくて読みきれないからとりあえずマージ」という空気が生まれがちです。第4章で見たとおり、AI は「渡されたテストを通すこと」に最適化する傾向があります。つまり AI 時代のコード品質は、レビューする人間の集中力ではなく、**ゲートとして機能するテストの質**で決まります。

GitHub のブランチ保護ルール（設定方法は公式ドキュメントを参照）で `test` ジョブを必須チェックに指定すれば、このルールは掛け声ではなくシステムによる強制になります。誰が書いたコードか ― 人間か、GitHub Copilot か、Claude Code か ― をゲートは区別しません。この平等さが重要です。

:::message
instructions ファイル（第5章）に「テストを実装に合わせて書き換えて通さない」と定めました。CI はこの規約の最後の防衛線ですが、AI がテスト自体を書き換えて通すことは技術的には可能です。**テストファイルの差分だけは人間が必ず目視する**運用をおすすめします。
:::

## カバレッジとの付き合い方：数字を目的にしない

`flutter test --coverage` が生成するカバレッジは、扱いを誤ると害になる指標です。本書のスタンスは明確です。

**カバレッジは診断ツールであって、目標ではありません。**

「カバレッジ90%以上を必須にする」と数字をゲートにした瞬間、数字を満たすためのテスト ― アサーションのないテスト、実装をなぞっただけのテスト ― が量産される傾向があります。カバレッジ100%でも、`expect` が正しい値を検証していなければ何も保証されません。逆に「count が 0 のとき CounterDecremented では何も emit されない」のようなテストは、行数はわずかでも仕様の急所を押さえています。

健全な使い方は2つです。**抜けの発見**（「`_onReset` のガード節が一度も通っていない」というテスト設計の穴に気づく）と、**傾向の監視**（数値が大きく下がった PR はテストなしの機能追加の兆候。絶対値でなく変化を見る）です。

AI との協働ではもうひとつ。AI に「カバレッジを上げて」と依頼すると、文字どおり数字を上げるだけのテストを書く傾向があります。依頼すべきは「この仕様のテストを書いて」であって、数字ではありません。

## AIによるPR作成とテストゲートの組み合わせ

AI コーディングツールは、いまやローカルでの補完にとどまらず、ブランチを切り、コードを書き、PR を作成するところまで実行できます。Claude Code にこう依頼したとしましょう。

> counter_bloc_test.dart に追加した3つのテストが通るように counter_bloc.dart を修正し、ブランチを切って PR を作成してください。テストファイルは変更しないでください。

このとき開発フローは次のループになります。

1. 人間がテストを書く（仕様の言語化）
2. AI が実装して PR を作る
3. CI がフォーマット・解析・テストを検証する
4. Red なら AI に差し戻す。Green になって初めて人間がレビューする

このループの美点は、**人間のレビューが始まる前に、機械的に検証できることはすべて終わっている**ことです。人間は低次の指摘から解放され、「この設計でよいか」「仕様の解釈は正しいか」という判断に集中できます。逆に、テストが貧弱なままこのフローを回すと「CI は緑だが仕様を満たしていない PR」が量産されます。ゲートの強度は、テストの解像度そのものです。

本章で、テストは「ローカルで実行するもの」から「マージ可否を決めるゲート」に昇格し、人間が仕様を定め、AI が実装し、CI が検証するという分業の全体像が揃いました。次章ではカウンターアプリを離れ、この開発スタイルを実務にどう持ち込むかを考えます。
