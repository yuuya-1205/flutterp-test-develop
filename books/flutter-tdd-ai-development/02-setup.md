前章では、テストを「AIへの仕様書」として使うテスト駆動AI開発の考え方を整理しました。本章では手を動かす準備として、開発環境と題材のコードを確認します。題材は Flutter プロジェクトを作成すると必ず付いてくる、あのデフォルトのカウンターアプリです。

## Flutter開発環境のセットアップ

Flutter SDK のインストール手順は OS ごとに異なるため、詳細は公式ドキュメントを参照してください。本書のコードは Dart 3 の機能（sealed class など）を使うため、Flutter は stable チャンネルの新しめのバージョンを想定しています。セットアップの確認には `flutter doctor` を使います。

```sh
flutter doctor
flutter create counter_app
cd counter_app
flutter test
```

最後の `flutter test` がポイントです。生成直後のプロジェクトでもテストが1件含まれており、この時点で緑（成功）になることを確認しておきます。テストが動く状態を最初に確保するのが、TDD の第一歩です。

なお、プロジェクト名は `counter_app` とします。以降のテストコードに登場する `import 'package:counter_app/...'` はこの名前が前提です。

次に、本書の後半（第6章以降）で使うパッケージをまとめて `pubspec.yaml` に追加しておきます。状態管理に BLoC、永続化に shared_preferences、テスト支援に bloc_test と mocktail を使います。

```yaml
dependencies:
  flutter:
    sdk: flutter
  bloc: ^9.0.0
  flutter_bloc: ^9.1.0
  equatable: ^2.0.7
  shared_preferences: ^2.3.0

dev_dependencies:
  flutter_test:
    sdk: flutter
  bloc_test: ^10.0.0
  mocktail: ^1.0.4
  flutter_lints: ^5.0.0
```

追加したら `flutter pub get` を実行しておきます。各パッケージの役割はそれぞれの導入章で説明するので、今は「後で使う道具を先に揃えた」と考えてください。

## AIコーディングツールの準備（GitHub Copilot / Claude Code）

本書では AIコーディングツールとして GitHub Copilot と Claude Code の2つを使います。どちらか一方でも本書の内容は実践できます。

- **GitHub Copilot**: エディタ内の補完とチャットが中心です。VS Code などに拡張機能としてインストールし、GitHub アカウントで認証します。カーソル位置の前後を文脈として読み、その場のコードを提案するスタイルが得意です。
- **Claude Code**: ターミナルで動くエージェント型のツールです。プロジェクト全体を読み、ファイルの作成・編集からテスト実行までを一連のタスクとしてこなします。「テストを渡して実装を任せる」という本書のワークフローと相性が良いツールです。

両者に共通する重要な仕組みが instructions ファイル（copilot-instructions.md / CLAUDE.md / AGENTS.md の総称）です。プロジェクトの規約を書いておくと AI が毎回それを前提に動いてくれるもので、第5章で詳しく設計します。この時点では「そういう仕組みがある」ことだけ覚えておけば十分です。

## デフォルトのカウンターアプリを読む：何が書かれているか

`flutter create` が生成する `lib/main.dart` の全文をここに載せる必要はありません。押さえるべき構造は次の3点です。

- `MyApp` がアプリのルートで、`MyHomePage` をホームとして表示する。
- `MyHomePage` の State クラスがフィールド `_counter` を持ち、`FloatingActionButton` のタップで `_incrementCounter` が呼ばれる。
- `_incrementCounter` の中で `setState` を呼び、`_counter++` して画面を再描画する。

つまり、**カウンターの「ロジック」（数を増やす）と「表示」（Text で描画する）が同じ Widget の中に同居している**状態です。動くコードとしては何の問題もありませんが、「数を増やすと1増える」というロジックだけを検証するテストが書きにくい構造です。この分離が第3章の最初の仕事になります。

もうひとつ、`test/widget_test.dart` も見ておきましょう。'Counter increments smoke test' という名前のテストが1件あり、内容は「起動時に 0 が表示され、『+』アイコンをタップすると 1 が表示される」ことの確認です。

ここで伏線をひとつ張っておきます。本書ではこのアプリを setState から BLoC へ作り替え、機能を大幅に増築しますが、**このデフォルトの widget_test.dart は1文字も変更せず、本書の最後まで緑のまま**です。なぜそれが可能なのか、そしてそれがなぜ重要なのかは、第6章で回収します。

## 本書で目指す最終形：機能拡張されたカウンターアプリの全体像

ゴールを先に共有しておきます。本書を通じて、このカウンターアプリは次のように育ちます。

- **デクリメント**: カウントを減らす。ただし 0 未満にはならない（第7章）
- **リセット**: カウントを 0 に戻す（第7章）
- **操作履歴**: 各操作の直前の値を履歴として記録する（第7章）
- **カウンターの複数化**: 独立した複数のカウンターを1画面に並べる（第7章）
- **永続化**: アプリを再起動してもカウントが復元される（第8章）
- **CI**: プルリクエストごとにテストが自動実行され、マージのゲートになる（第9章）

機能の一つひとつは小さいものです。しかし「テストを先に書き、実装を AI に任せる」サイクルを繰り返しながら増築していくと、カウンターという最小の題材でもテスト駆動AI開発の要点を一通り体験できます。

環境と題材、そしてゴールが揃いました。次章では AI をいったん脇に置き、まず人間だけの手で TDD のサイクルを回してみます。
