title: "第4章 AIをサイクルに入れる ― テストを書いて、実装はAIに"
---

前章では、人間だけの手で Red / Green / Refactor のサイクルを一周し、`Counter` クラスとそのテストを完成させました。本章ではいよいよ、このサイクルに AI を組み込みます。鍵となるのは「どの工程を AI に渡し、どの工程を人間が握り続けるか」という分業の設計です。

## 人間がRed（テスト）、AIがGreen（実装）という分業

本書が提案する分業は、次のシンプルな一行に集約されます。

**人間が Red（失敗するテスト）を書き、AI が Green（テストを通す実装）を書く。**

なぜこの分け方なのでしょうか。TDD のサイクルを工程ごとに見直すと、それぞれ求められる能力が異なることに気づきます。

- **Red**: 「何を作るべきか」を決める工程です。仕様の決定であり、価値判断です。ここを誤ると、あとの工程がどれだけ正確でも意味がありません。
- **Green**: 「決まった仕様を満たすコードを書く」工程です。ゴールがテストという実行可能な形で明示されているため、正解かどうかを機械的に判定できます。
- **Refactor**: 「テストを緑に保ったまま内部を改善する」工程です。こちらも、テストという安全網の内側で行われます。

AI コーディングツール（GitHub Copilot や Claude Code）が最も力を発揮するのは、**ゴールが明確で、達成を機械的に検証できるタスク**です。Green はまさにその条件を満たしています。テストがあれば、AI の出力が正しいかどうかを人間が目視でレビューし切れなくても、`flutter test` の結果が代わりに判定してくれます。

一方 Red は、プロダクトとして「何が正しいのか」を決める工程です。ここは人間が責任を持つべき最後の砦であり、同時に、AI への最良のプロンプトを作る工程でもあります。第1章で述べた「テストは仕様書である」という言葉は、AI 開発の文脈では文字どおりの意味を持ちます。テストは自然言語のあいまいさを排した、実行可能な仕様書だからです。

## テストだけを渡したとき、AIは何を作るか：実験

分業の考え方が机上の空論でないことを確かめるため、小さな実験をしてみましょう。前章で書いた `test/counter/counter_test.dart` だけを AI に渡し、実装ファイルは白紙の状態から書かせます。

Claude Code や GitHub Copilot のチャットに、たとえば次のように依頼します。

> 以下のテストがすべて通るように lib/counter/counter.dart を実装してください。テストは変更しないでください。
>
> （test/counter/counter_test.dart の内容を貼り付け）

このとき AI が返してくる典型例は、前章で私たちが自分の手で書いたものとほぼ同じ形になります。

```dart
// lib/counter/counter.dart
class Counter {
  int _value = 0;

  int get value => _value;

  void increment() => _value++;
}
```

興味深いのは、テストの中に「フィールドを持て」とも「`_value++` を使え」とも一言も書いていないのに、この形に収束することです。「初期値は 0」「1回呼ぶと 1」「2回呼ぶと 2」という3つの制約を同時に満たそうとすると、内部に可変の状態を持つ以外の選択肢が実質的になくなるためです。テストの制約が十分に強ければ、実装は自然と意図した形に導かれる。これがテストを仕様書として使う、ということの具体的な手触りです。

なお、AI の出力は確率的であり、実行のたびに細部（`_value++` か `_value = _value + 1` か、など）は揺れます。本書で示す出力はあくまで典型例であり、手元の環境で一字一句同じ結果になるとは限りません。だからこそ「出力をテストで検証する」仕組みが重要になるのです。

## 曖昧なテストは曖昧なコードを生む：仕様書としてのテストの解像度

前節の実験がうまくいったのは、テストが仕様を過不足なく記述していたからです。では、テストの解像度が低かったらどうなるでしょうか。

仕様書としてのテストには「解像度」があると本書は考えます。解像度とは、**そのテストを通す実装の集合が、どれだけ意図した実装に絞り込まれているか**の度合いです。

- 解像度が高いテスト: テストを通す実装は、実質的に意図した挙動を持つものに限られる。
- 解像度が低いテスト: 意図とかけ離れた実装でも通ってしまう「抜け道」が残っている。

人間のペアプログラミング相手なら、テストに書かれていない意図を常識や文脈で補ってくれることがあります。しかし AI は、良くも悪くも**与えられたゴールを最短距離で達成しようとする**傾向があります。テストに抜け道があれば、その抜け道を通った実装が返ってくることは十分にあり得ます。AI が意地悪をしているのではなく、こちらの仕様書が粗かっただけです。

:::message
「AI がテストだけ通す手抜き実装を返してきた」と感じたときは、AI を責める前にテストを疑ってください。多くの場合、それはテストが仕様を語り切れていないというシグナルです。
:::

## 悪いテスト/良いテストで生成結果を比較する

解像度の違いを、具体的なコードで比較してみましょう。まず「悪いテスト」の典型例です。

```dart
test('カウントできる', () {
  final counter = Counter();
  counter.increment();
  expect(counter.value, greaterThan(0));
});
```

一見それらしいテストですが、仕様書として読むと穴だらけです。テスト名「カウントできる」は操作と結果を語っておらず、アサーションも「0 より大きければ何でもよい」としか言っていません。初期値がいくつであるべきか、2回呼んだらどうなるべきかにも触れていません。

このテストだけを渡して「通る実装を書いてください」と依頼したとき、AI が返しがちな典型出力がこちらです。

```dart
class Counter {
  int get value => 1; // テストは通るが、意図とは違う

  void increment() {}
}
```

`value` は常に 1 を返し、`increment()` は何もしません。それでも `greaterThan(0)` は満たされるので、テストは緑になります。仕様書に「常に 1 を返す実装でも合格」と書いてあったのだから、AI としては要求どおりの納品です。

次に「良いテスト」、つまり前章で書いた3ケースを渡した場合と比べてみます。

```dart
// test/counter/counter_test.dart
import 'package:flutter_test/flutter_test.dart';

import 'package:counter_app/counter/counter.dart';

void main() {
  group('Counter', () {
    test('生成直後の value は 0 である', () {
      // given
      final counter = Counter();

      // then
      expect(counter.value, 0);
    });

    test('increment で value が 1 増える', () {
      // given
      final counter = Counter();

      // when
      counter.increment();

      // then
      expect(counter.value, 1);
    });

    test('increment を2回呼ぶと value は 2 になる', () {
      // given
      final counter = Counter();

      // when
      counter.increment();
      counter.increment();

      // then
      expect(counter.value, 2);
    });
  });
}
```

「初期値は 0」というケースがあるので、定数 1 を返す実装は即座に落ちます。「1回で 1」「2回で 2」の2ケースがあるので、`increment()` が状態を変化させない実装も落ちます。3つのケースが互いの抜け道をふさぎ合い、実状態を持つ実装以外が生き残れなくなっているのです。この解像度なら、AI の出力は前節で見たような実装に安定して収束します。

2つのテストを見比べると、良い仕様書としてのテストの条件が浮かび上がります。

- **境界と初期値を明示する**: 「生成直後は 0」のような当たり前に見える事実こそ、抜け道をふさぐ最初の一手になります。
- **アサーションは具体的な値で**: `greaterThan(0)` のような幅のある検証は、実装にも幅を許します。意図が「1 になる」なら `expect(counter.value, 1)` と書きます。
- **ケースを重ねて挙動の変化を語る**: 1回と2回のケースをセットにすることで、「呼ぶたびに増える」という時間方向の仕様が伝わります。
- **テスト名が仕様の文になっている**: 「increment で value が 1 増える」という名前は、そのまま仕様書の一文として読めます。「カウントできる」では何も語っていません。

人間だけの TDD でも、これらはテストの品質として重要でした。テスト駆動AI開発では、それがそのまま**プロンプトの品質**に直結します。テストの解像度を上げることは、AI への指示の解像度を上げることと同じなのです。

## リファクタリングもAIに任せてよいか：テストが担保するもの

残る工程は Refactor です。結論から言えば、**リファクタリングも AI に任せられます。ただし、テストが担保する範囲を理解した上で**、という条件付きです。

リファクタリングの定義は「外から見たふるまいを変えずに内部構造を改善すること」でした。「ふるまいが変わっていないこと」の判定装置がテストです。テストが十分な解像度で書かれていれば、AI がどれだけ大胆にコードを書き換えても、緑のままである限り仕様は守られています。依頼するときは、その制約を明示します。

> lib/counter/counter.dart をリファクタリングしてください。test/counter/counter_test.dart のテストはすべて通ったままにすること。テストコード自体は変更しないでください。

「テストを変更しない」という一文は必ず入れてください。AI は「すべてを緑にする」というゴールに向かうとき、実装ではなくテストの側を書き換えて帳尻を合わせようとすることがあるからです。テストは仕様書ですから、実装の都合で書き換えられては本末転倒です。

一方で、テストが担保**しない**ものも正しく認識しておく必要があります。

- **テストに書かれていないふるまい**: たとえば現状のテストは `value` が負になるケースに触れていません。リファクタリング中に AI がその領域の挙動を変えても、テストは検知できません。
- **可読性・設計の良し悪し**: 読みにくいコードでもテストは通ります。命名や構造の妥当性の最終判断は人間のレビューの仕事です。
- **性能**: 計算量が悪化しても、結果が同じならテストは緑のままです（性能が仕様なら、それもテストにする必要があります）。

つまり Refactor は「AI が書き換え、テストがふるまいを守り、人間が設計を見る」共同作業になります。任せる範囲を広げたければ、担保したい性質をテストに書き足していく。ここでも主導権はテスト、すなわち人間の側にあります。

---

本章では、人間が Red を書き AI が Green を書くという分業を、悪いテストと良いテストの生成結果の違いを通して確かめました。テストの解像度がそのまま AI への指示の解像度になる、というのが本章の核心です。しかし、テストだけでは伝えきれないものもあります。ディレクトリ構成、命名規約、チームの流儀。次章では、それらを AI に教えるための instructions ファイルの設計を扱います。
